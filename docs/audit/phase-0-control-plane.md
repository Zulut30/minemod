Дата доступа: 2026-07-21. Этот документ фиксирует только Phase 0 scaffold; он не означает, что production-возможности из общего плана уже реализованы. Оригинальные implementation, документация и schemas репозитория выпускаются под Apache-2.0 согласно [ADR-0001](../decisions/0001-product-and-output-licensing.md); лицензии ниже относятся только к соответствующим upstream-компонентам.

## Решения и ограничения

### Frozen schema contract Phase 0

- Wire-версия обоих строгих контрактов фиксируется как числовое `schemaVersion: 0`. Числовой `schemaVersion: 1` в YAML-примере исследовательского плана является иллюстрацией будущего production IR, а не совместимым alias Phase-0 v0; строка `"0"` также отклоняется. Миграция к v1 должна быть явной и версионированной.
- ModSpec v0 выражает canonical разделы `gameplay.items/blocks/entities/recipes/summoning/screens`, `assets.models/textures/animations`, `integrations.jei/jade` и `tests.gameTests`; generic public `resources[]` отсутствует. `assets.artSpec` — безопасный относительный `.json/.yaml/.yml` path, dependency entries — bounded lowercase mod ids (`geckolib`, `jei`, `jade`), а `auto` integrations обязаны находиться только в `dependencies.optional`. Typed semantic validation различает item/block/entity/model reference domains, требует serializer для custom recipe и server validation для screen.
- `validateSpec(value, expectedKind = "auto", options = {})` и `validateInlineSpec(payload, expectedKind = "auto", options = {})` по умолчанию loader-neutral: валидные NeoForge, Fabric, Forge и Paper targets принимаются, а animation asset сам по себе не навязывает loader-specific dependency. Единственный bounded opt-in profile `neoforge-26.1.2-java-25` требует Minecraft 26.1.2, NeoForge и Java 25; если `assets.animations` непуст, только этот profile дополнительно требует bare mod id `geckolib` именно в `dependencies.required`. Maven coordinate или optional dependency не удовлетворяют этому правилу. Trusted programmatic API принимает `options.profile`, а CLI включает его только через `--profile`. MCP намеренно не публикует и не передаёт validation options: его tool всегда вызывает loader-neutral validation только с `kind` и `payload`.
- ArtSpec v0 требует `assetClass`, bounded non-empty `targetMatrix`, class-specific `targetContexts`, palette/hue-value hierarchy, lighting source/direction, outline/saturation/detail/material rules, silhouette language, forbidden references, texel density, reference rights/provenance policy и полные budgets. Metrics и budgets содержат triangles. Target tuple фиксирует Minecraft, loader+version, Java и согласованный `runtime.version`, renderer id+version; regression fixture baseline — NeoForge `26.1.2.80`, Java `25`, vanilla renderer `26.1.2`. Повтор tuple и case-insensitive повтор palette color отклоняются на более позднем pointer.
- Safe asset allowlist включает `.png`, `.json`, `.bbmodel`, `.nbt`, `.snbt`, `.schem`, `.schematic` и `.blend`, чтобы классы rubric (`structure`, `decorative-mesh`) были представимы без traversal. Это не означает, что Phase 0 уже реализует class/codec/runtime validation: эти проверки, preview evidence и human release gate остаются Phase 3 и manual [Art Quality Rubric v0](../quality/art-quality-rubric-v0.md).

- `packageManager` фиксирует pnpm 11.8.0 официальной Corepack-формой `pnpm@11.8.0+sha512.c1f5…55f26`; hex digest совпадает с registry SRI `sha512-wfXn…VfJg==` для `pnpm-11.8.0.tgz`. Оба полных значения записаны в machine-readable provenance и проверяются тестом.
- `engines` фиксирует точный Node 24.11.0 и pnpm 11.8.0, а workspace `engineStrict: true` превращает несовпадение в ошибку. Негативный regression запускает frozen offline install с детерминированным `node-version=22.22.2` и требует `ERR_PNPM_UNSUPPORTED_ENGINE`.
- `pnpm install --frozen-lockfile` является immutable gate: несоответствие manifest/lockfile должно завершать установку ошибкой.
- `verifyDepsBeforeRun: false` запрещает script-командам неявно запускать изменяющий dependency install; install всегда выполняется отдельным явным gate.
- `eslint . --max-warnings 0` является самостоятельным lint gate на pinned ESLint/typescript-eslint и не подменяется TypeScript compilation.
- TypeScript gate запускает `tsc --project tsconfig.json`; project `include` автоматически покрывает каждый `apps/**/*.ts`, `packages/**/*.ts` и `fixtures/**/*.ts`, включая production, tests и fixture helpers. Явного root-file списка, который мог бы молча пропустить новый файл, нет; script и include защищены regression-тестом.
- Workspace overrides удерживают внутренние `@typescript-eslint/types` и `@typescript-eslint/tsconfig-utils` на проверенной линии 8.63.0: upstream использует caret ranges, которые иначе подтягивали бы только что опубликованные 8.65.0 и требовали bypass minimum-release-age policy.
- Transitive Hono закреплён на совместимой с MCP SDK версии 4.12.25 вместо уязвимой 4.12.19: `pnpm audit` помечает версии ниже 4.12.25 как high по GHSA-88fw-hqm2-52qc. HTTP/CORS path не используется stdio-only entrypoint, но patched pin устраняет известную уязвимость из dependency graph, а не полагается только на недостижимость.
- Transitive body-parser закреплён на 2.3.0: Express 5.2.1 допускает `^2.2.1`, поэтому patch для low GHSA-v422-hmwv-36x6 остаётся внутри объявленного upstream range.
- После этих pins `pnpm audit --audit-level high` завершается успешно и известных high advisories нет. Это не означает zero advisories: остаётся moderate GHSA-v2v4-37r5-5v8g в `@modelcontextprotocol/sdk -> express-rate-limit@8.4.1 -> ip-address@10.1.0`; исправление опубликовано в `ip-address>=10.1.1`, но express-rate-limit объявляет точную зависимость `10.1.0`. Принудительный override нарушил бы upstream constraint. Phase 0 импортирует только stdio MCP path, не HTTP/rate-limit graph и не уязвимые HTML-emitting методы Address6; риск принят временно до совместимого upstream graph и записан machine-readable в provenance.
- Zod schema — единственный источник ModSpec/ArtSpec: из тех же объектов выводятся TypeScript types, runtime validation и draft 2020-12 JSON Schema. Это исключает ручную рассинхронизацию generated schema.
- Для всех иначе Unicode-capable строк v0 действует единая BMP-only политика: допустимы BMP scalar values, включая не-ASCII, но supplementary pairs и lone surrogates отклоняются общим pattern. Поэтому Zod UTF-16 `min`/`max` и JSON Schema code-point `minLength`/`maxLength` эквивалентны; emitted schemas проверяются на те же `maxLength` и pattern.
- ResourceLocation и asset destination состоят только из непустых lowercase segments; `.`/`..`, `//`, trailing slash, absolute path и backslash запрещены, точки внутри обычного segment разрешены. Namespace `.`/`..` также запрещён. ResourceLocation uniqueness проверяется внутри фактического registry/resource domain, а не глобально: Block и BlockItem могут иметь одинаковый id, как и model/texture/animation или JEI/Jade ids; повтор внутри `gameplay.items`, `gameplay.blocks`, каждого другого gameplay registry, `assets.models`, `assets.textures`, `assets.animations`, GameTests или ArtSpec assets отклоняется на later-entry pointer. Canonical asset destinations по-прежнему дедуплицируются между всеми ModSpec asset sections и отдельно в ArtSpec, потому что один output path не может безопасно получить два содержимых. Dependency mod ids образуют одну ordered uniqueness domain: повторы внутри `required`, внутри `optional` и пересечение дают `DUPLICATE_DEPENDENCY`.
- Все diagnostic paths являются RFC 6901 JSON Pointers: document root — пустая строка, а unknown object keys разворачиваются по одному, сортируются и экранируют `~`/`/` в полном parent+key pointer.
- Zod 4.4.2 не сообщает own enumerable JSON key `__proto__` как unknown key. После own-data normalization, но до semantic validation/Zod, отдельный bounded deterministic pass явно отклоняет этот reserved key на любом object level как `SCHEMA_INVALID`. Root, nested project и object внутри asset array создаются через `JSON.parse` (не object-literal prototype semantics), проверяют точные RFC 6901 paths и подтверждают intent emitted schemas: ключ отсутствует в `properties`, а соответствующие objects имеют `additionalProperties:false`.
- CLI и MCP имеют независимые package entrypoints и dependency boundaries. Production CLI принимает только help/version/spec-validate grammar, включая точную optional profile form; test harness и subprocess fixtures вынесены в отдельный `apps/cli/test.ts`, а ModSpec/validation assertions и fixtures — в dedicated `packages/*/test.ts` и `fixtures/specs/**`. Поэтому скрытого production `--self-test`, fixture exports и test-only `node:child_process`/assert imports в production packages нет. MCP использует только локальный stdio transport и регистрирует один bounded tool `mcdev_spec_validate`. Initialize regressions явно подтверждают отсутствие advertised resources/prompts; eval, shell, network, publish и запись файлов отсутствуют.
- MCP tool input зарегистрирован как Zod `strictObject`; `tools/list` публикует ровно public properties `kind`/`payload` и `additionalProperties:false`. Любой `profile` — включая точный trusted programmatic profile — является для MCP неизвестным attacker-controlled argument и возвращает официальный SDK 1.29 `CallToolResult` с `isError:true` до вызова handler. Zod 4.4.2 и generic MCP JSON-RPC schema специально пропускают/удаляют exact own key `__proto__`, поэтому bounded stdio boundary сначала выполняет `JSON.parse` уже ограниченного frame и проверяет `Object.hasOwn(params.arguments, "__proto__")`. Guard не сериализует attacker-controlled parsed values повторно: он создаёт малый canonical `tools/call` surrogate с зарегистрированным tool name, strict-unknown sentinel `__proto_x` и только допустимым scalar request id; notification без id остаётся некоррелированной. Размер surrogate повторно проверяется до forwarding. Поэтому compact JSON numbers (`1e20`) не могут расширить near-cap input выше transport cap, SDK возвращает `isError:true` с исходным id, а handler-count не меняется. Raw regressions покрывают literal и `\u005f_proto__` spellings, nested own key, one-byte fragmentation, rejected public `profile`, payload string с текстом `__proto__` и exact 2097137-byte exponent-amplification frame. In-schema optional refinement остаётся дополнительной защитой, если исходный parsed object достигает tool schema напрямую; sentinel не публикуется в JSON Schema.
- Protocol regression test подключает официальный MCP `Client` через linked `InMemoryTransport`, проверяет initialize, `tools/list`, `tools/call` и единственность tool; отдельные raw stdio subprocess tests доказывают, что stdout содержит только JSON-RPC frames и что процессы корректно закрываются.
- Перед SDK `StdioServerTransport` установлен bounded Transform: он не передаёт SDK неполный frame и ограничивает каждый newline-delimited JSON-RPC frame двумя MiB до SDK buffering/`JSON.parse`. Один preallocated cap-sized Buffer делает retained allocation независимым от attacker-controlled fragmentation; на complete frame копируются только фактические bytes. Два MiB покрывают максимальный inline payload даже при шестикратном outer JSON escaping управляющих символов; fragmented valid frame сохраняется побайтно, а complete/unterminated frame размером cap+1 завершают процесс fail-closed. Это byte/frame bound, а не заявление об общем DoS-hardening JSON parser.
- Максимальный inline payload — 262144 UTF-8 bytes. Для raw JSON и прямого API до любых semantic reads и Zod `safeParse` итеративный structural preflight ограничивает depth, total JSON nodes, enumerable own object keys, длину одного/суммарную длину ключей, любой неизвестный array и каждую известную ModSpec/ArtSpec collection согласно её schema maximum. После проверки plain/null prototype и Proxy rejection enumerable own string keys перечисляются лениво через `for...in`: object прекращается на 65-м own key до построения/sort полного списка, array — на первом enumerable non-index own extra, а первый inherited enumerable candidate немедленно отклоняется. Поэтому загрязнённые стандартные `Object.prototype`/`Array.prototype` не обходят JS-level traversal bounds. Значения читаются только через own data descriptors без вызова getters и копируются в detached normalized graph с null-prototype objects; semantic validator и Zod не читают исходный graph или prototype chain. Non-enumerable и symbol metadata находятся вне JSON data model, сознательно игнорируются и никогда не попадают в normalized graph; `Reflect.ownKeys` не вызывается. Custom prototypes, inherited-only graphs, enumerable accessors, sparse/enumerable-extra arrays, cycles, `undefined`/bigint/function и non-finite numbers отклоняются. Preflight останавливается на первом нарушении с единственным детерминированным RFC 6901 diagnostic. Adversarial regressions создают отдельно object и array с 250000 enumerable own extra keys, а изолированные subprocess regressions — по 250000 enumerable inherited keys на стандартных Object/Array prototypes; на pinned Node 24 все требуют rejection менее чем за 2 секунды с менее чем 16 MiB retained RSS после GC, нулём getter calls и нулём вызовов instrumented semantic duplicate handler. Raw/API/MCP regressions используют 40000 пустых `gameplay.items` в payload ниже byte cap и доказывают, что запрос отклоняется как `STRUCTURE_LIMIT_EXCEEDED`, а не создаёт затем десятки тысяч Zod issues; API regressions отдельно покрывают inherited amplification и getter bombs с нулём вызовов. Одновременно ModSpec/ArtSpec с каждой collection на её maximum проходят под неизменными global bounds 16384 nodes, 8192 object keys и 131072 key chars. Повторы IDs внутри каждого gameplay/asset/test registry, canonical asset paths, dependency mod ids, target tuples, palette colors и typed references проверяются с точными JSON Pointers; совпадение id между независимыми registries не является ошибкой.
- Node binaries следует проверять по подписанному `https://nodejs.org/dist/v24.11.0/SHASUMS256.txt.asc`. Для npm artifacts источником целостности служат `resolution.integrity` lockfile; provenance manifest не подменяет сам lockfile.

## Лицензионная граница

MIT/Apache-2.0 лицензии зависимостей совместимы с разработкой scaffold, но не изменяют Apache-2.0 лицензию оригинальных материалов продукта и не разрешают копирование кода или assets из reference-модов. Политика generated output и границы third-party/Minecraft материалов определены в [ADR-0001](../decisions/0001-product-and-output-licensing.md). Publish остаётся отдельной, отсутствующей операцией.
